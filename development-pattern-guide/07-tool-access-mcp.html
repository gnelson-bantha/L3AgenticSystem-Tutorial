Chapter 7

Tool Access Through a Protocol Boundary

Giving agents the tools they need over the Model Context Protocol instead of hard-wiring functions into them.

The Boundary

Why Tools Live in Another Process

The selector agent just called some query tools; here's why those tools deliberately live across a wire.

In the last chapter the selector agent asked which targets an event affects, and it answered by calling tools. The obvious way to give an agent a tool is to hand it a C# delegate: write a method, register it as a function, done. That works, and for a quick prototype it's fine. This system does something deliberately heavier: it puts the tools in a separate server and has the agents reach them over the Model Context Protocol. The extra ceremony buys a boundary, and the boundary is the point.

MCP is a standard protocol for exposing tools to agents. One side is a server that publishes a catalog of tools; the other side is a client (an agent) that discovers and calls them, without knowing or caring where they run or how they're implemented. Because that contract is a protocol rather than a method signature, the tools can live in another process, be written by another team, be swapped for a real data backend, or be reused by something that isn't even an agent. None of those changes touch the agents. That is the freedom you're buying.

Independent change

Tools evolve behind the protocol. Swap a stub data source for a production database, add a new query, fix a bug, and the agents keep calling the same named tools, unchanged and unaware.

Shared reuse

The same tool server feeds the selector agent, the research agent, and the external conversational front door. One catalog, many consumers, no duplication.

A place to govern

A boundary is a natural checkpoint. It's where you decide which callers see which tools, which is exactly the filtering trick at the end of this chapter.

The Server

A Tool Server in Practice

It's an ordinary ASP.NET host that publishes a few annotated methods as tools.

The tool server is its own small project. It uses the MCP server package for ASP.NET, picks the Streamable HTTP transport (a long-lived HTTP connection agents can stream over), and registers a class whose methods are the tools. There's almost no ceremony: annotate the host, annotate the methods, map the endpoint.

var builder = WebApplication.CreateBuilder(args);

builder.Services
    .AddMcpServer()
    .WithHttpTransport()          // Streamable HTTP: a durable connection to stream over
    .WithTools<TargetQueryTools>();

var app = builder.Build();
app.MapMcp();                     // publish the MCP endpoint
app.Run();

The tools themselves are just methods with a couple of attributes. Each one gets a description, and that description is not decoration: it's what the model reads to decide when to call the tool, so it's written for the model's benefit. The sample exposes a small family of target queries (all targets, by region, by size, by category, and a combined region-and-size query), each backed by the entity store.

[McpServerToolType]
public sealed class TargetQueryTools(EntityStore store)
{
    [McpServerTool, Description("List every tracked target.")]
    public Task<IReadOnlyList<TargetProfile>> QueryAllTargets()
        => store.GetAllAsync();

    [McpServerTool, Description("Find targets located in a given region.")]
    public Task<IReadOnlyList<TargetProfile>> QueryTargetsByRegion(string region)
        => store.GetByRegionAsync(region);

    [McpServerTool, Description("Find targets at or above a minimum size.")]
    public Task<IReadOnlyList<TargetProfile>> QueryTargetsByMinSize(decimal minSize)
        => store.GetByMinSizeAsync(minSize);
}

That's the whole server. Notice what's not here: nothing about agents, nothing about the workflow, no awareness of who's calling. The server's only concern is "here are some tools and what they do." Keeping it that ignorant is what lets it be reused so freely.

The Client Side

Handing Tools to an Agent

On the agent side, a discovered MCP tool is just another function the model can call.

An agent connects to the server, asks what tools it offers, and binds them. The connection is wrapped in a small resilient helper so that if the link drops (servers restart, networks blip) it reconnects rather than failing the run. That resilience matters because these are long-lived, autonomous runs; you don't want a transient hiccup to sink an hour of work.

// A reconnecting wrapper over the MCP connection, so a dropped link self-heals
// instead of killing a long-running autonomous run.
var connection = new ResilientMcpConnection(toolServerUri);
await connection.ConnectAsync();

IList<McpClientTool> tools = await connection.ListToolsAsync();

The elegant part is the last step. A discovered tool comes back as an McpClientTool, which derives from the framework's AIFunction type. That means a remote tool, living in another process and reached over a protocol, drops into an agent in exactly the same place a local C# function would. The agent can't tell the difference, and doesn't need to.

// McpClientTool : AIFunction, so remote tools bind exactly like local ones.
var selector = new ChatClientAgent(client, new ChatClientAgentOptions
{
    Name = "SelectorAgent",
    Instructions = selectorInstructions,
    ChatOptions = new ChatOptions { Tools = [.. tools] }
});
Least Privilege

Hiding the Controls With a Tool Filter

The server publishes more than the agents should see, so the agents get a filtered view.

Here's where the boundary earns its keep a second time. The same tool server also publishes a few workflow-control tools (submit an event, get a run's status, list runs) because the external conversational front door needs them to drive the system from the outside. But the internal agents must never call those. A research agent that could submit new events or poke at workflow status is a research agent that can wander off the rails, or worse, drive the system in a loop.

The fix is a filter applied on the client side, before the tools are bound to an internal agent. The agent simply never receives the control tools, so it can't call what it can't see.

// The control tools belong to the external front door, not the internal agents.
// Filter them out before binding, so an internal agent literally cannot see them.
var dataTools = tools
    .Where(McpToolFilter.ExcludeWorkflowControlTools)
    .ToList();

var research = new ChatClientAgent(client, new ChatClientAgentOptions
{
    Name = "ResearchAgent",
    Instructions = researchInstructions,
    ChatOptions = new ChatOptions { Tools = [.. dataTools] }
});
Least privilege for agents

An autonomous agent will use any tool you give it, which means the tools you give it are its permissions. Publishing a broad catalog at the server and then handing each agent only the slice it needs is least privilege applied to agents. The protocol boundary is what makes that slicing possible, because filtering happens at the seam rather than being tangled into agent logic.

One more thing this boundary enables, worth a sentence before we move on: because tools are just an MCP catalog, you can point an agent at additional servers. The sample can optionally attach a separate analytics data agent exposed over its own MCP endpoint, and the binding code looks identical. We'll revisit that in the extending chapter.

The selector and research agents now have their data. The work items the selector produced are sitting in the storage bridge, untouched. That bridge is the structural heart of the whole system, and it's finally time to give it the chapter it deserves.
