Chapter 15

Extending the System

The bridges you built for decoupling are the same bridges you extend through. Here's where each one opens up.

Where to Push

The Decouplings Are the Extension Points

Every boundary the pattern drew to keep things apart doubles as a place to add something new.

Now that we understand the whole pattern, we can identify the extension points. Each of the boundaries we've established in the pattern serves as an extension point. Dependency injection at the composition root is where you add a participant. The protocol boundary in front of tools is where you add a capability. The storage bridge is where you change how work is dispatched. The middleware chain is where you add a cross-cutting concern. The workflow graph is where you change how the system is triggered. None of these required foresight about this specific extension.

That's the benefit of decoupling. You don't extend this system by editing its core. You extend it by adding a part at a boundary and letting the existing composition pick it up. Let's walk the boundaries one at a time and see what each lets you add.

More Hands, More Tools

Adding Agents and Tools

A new collaborator joins the participant list; a new capability joins the tool server.

Adding an agent is fairly simple. You build it through the same pool (AgentPool) that builds the others, so it inherits the entire middleware stack (retry, identity baggage, governance, telemetry, compaction) for free, and then you add it where it needs to be, in the agent call order, to the Group Chat builder. The collaboration's natural-language coordination from Chapter 9 absorbs the newcomer without special wiring. The manager simply has another participant to orchestrate.

// New agent: built by the same pool, so it gets the full middleware chain for free.
var factCheckAgent = agentPool.Build("FactCheckAgent", factCheckInstructions);

var workflow = AgentWorkflowBuilder
    .CreateGroupChatBuilderWith(manager)
    .AddParticipants(researchAgent, generatorAgent, reviewerAgent, factCheckAgent)
    .Build();

A new tool is just as contained. Because tools live behind the MCP boundary, you add one by writing a method on the tool server and marking it, and every agent that's allowed to see it can call it through the same protocol. The agents don't change. Their menu of capabilities does. And when a capability is substantial in its own right (say, analytics over a warehouse), you don't have to cram it into the existing server at all. You can stand up a separate data agent (for example, a Microsoft Fabric data agent) that exposes its own MCP endpoint, and point a client at it exactly the way Chapter 7 connected to the first one. The protocol boundary makes "another tool" and "another whole tool service" the same shape of change.

// New tool: one annotated method on the tool server; agents reach it over MCP unchanged.
[McpServerTool, Description("Look up the canonical record for a given identifier.")]
public Task<string> LookupRecordAsync(string identifier) =>
    _records.GetCanonicalAsync(identifier);
New Engines, New Doors

Swapping Backends and Triggers

Change what powers an agent, or change what wakes the system up, without touching the loop in between.

The agents are coded against an abstraction, not a specific model host, so swapping in a different backend is a composition-root change rather than a rewrite. A hosted agent defined in Azure AI Foundry, for instance, comes in through AIProjectClient.AsAIAgent(...) against an agent reference, and then flows through the very same middleware extensions as a locally constructed agent. The collaboration loop never learns whether a participant is backed by a direct model client or a Foundry-hosted definition. It just sees an agent.

// Swap the engine: a Foundry-hosted agent, wrapped in the identical middleware chain.
var hosted = projectClient
    .AsAIAgent(new AgentReference(hostedAgentId))
    .AsBuilder()
    .WithRateLimitRetry()
    .WithAgent365Baggage()
    .WithOpenTelemetry()
    .Build();

Changing how the system is triggered is the other big lever that lives at the front of the intake graph. The reference system is woken by an operator submitting events through a conversational front door, but nothing downstream depends on that. You could replace it with a continuous scanner that watches a feed and emits the same normalized events, and the intake workflow, the bridge, and the processing stage wouldn't notice. Similarly, the storage-bridge dispatch from Chapter 8 was deliberately built so the polling mechanism could be swapped for a managed queue (Azure Event Grid or Azure Service Bus) when you outgrow polling. In every case the bridge absorbs the change: a new trigger produces the same work, a new dispatcher moves the same work, and the autonomous middle stays put.

New Trigger

Replace the conversational front door with a continuous scanner or scheduled job that emits the same normalized events. Intake onward is unaffected.

New Dispatch

Swap the blob-polling bridge for Event Grid or Service Bus when scale demands it. The claim-and-process contract stays the same.

New Backend

Bring agents from a different host (a Foundry-defined agent) in through the same builder, wrapped in the same middleware.

The Uniform Bridge

Extending the Middleware Itself

A new cross-cutting concern is one decorator and one extension method away.

The most reusable extension point of all is the middleware chain, because adding to it follows a single recipe no matter what you're adding. You write a class that wraps the agent's chat client (deriving from the framework's delegating base), do your work before or after delegating to the inner client, and expose a small .With... extension so it reads like the rest of the chain. Want a per-tenant budget guard, a prompt-injection screen, a redaction pass? Each is the same shape, and each slots into the chain wherever its order belongs, just as you saw in Chapter 11.

// A new cross-cutting concern: wrap, act, delegate, expose a .With... extension.
public sealed class BudgetGuardChatClient(IChatClient inner, IBudget budget)
    : DelegatingChatClient(inner)
{
    public override async Task<ChatResponse> GetResponseAsync(
        IEnumerable<ChatMessage> messages, ChatOptions? options, CancellationToken ct)
    {
        budget.EnsureWithinLimit();                 // act before
        return await base.GetResponseAsync(messages, options, ct);  // then delegate inward
    }
}

public static class BudgetGuardExtensions
{
    public static ChatClientBuilder WithBudgetGuard(this ChatClientBuilder b, IBudget budget) =>
        b.Use(inner => new BudgetGuardChatClient(inner, budget));
}

So the system extends in five congruent ways, and all of them are additions at a boundary rather than surgery on the core: a participant at the composition root, a capability at the protocol boundary, a dispatcher at the storage bridge, a trigger at the head of the graph, and a concern in the middleware chain. If you internalize just one thing from this chapter, let it be that the cost of extension is set long before you extend, by how cleanly you drew your bridges.

There's one practical question left that we've been gesturing at the whole time: how do you actually run this, on a laptop and in the cloud, without maintaining two different versions of the truth? Let's move on to find out!
