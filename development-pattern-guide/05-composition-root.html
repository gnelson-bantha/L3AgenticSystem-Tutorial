Chapter 5

The Composition Root: Orchestrating the System

Where the whole system gets constructed, wired together, and handed its dependencies before any work begins.

First Principles

Where a Distributed System Begins

The architecture map showed the components; this chapter shows the one place where those components get built.

Every program has a spot where the abstract design becomes concrete objects: dependencies get constructed, handed to whatever needs them, and the application starts running. That spot has a name, the composition root, and keeping it in one place is one of the highest-leverage decisions you can make. When construction lives at the root and everything else just receives what it needs, the rest of the code stays ignorant of how things are built. That ignorance is what makes the pieces swappable, which is the entire point in a system you want to extend later.

A distributed system has this at two levels, and it's worth learning about both. There's the composition root for each individual service (its Program.cs, where that one process wires up its own dependencies), and there's a composition root for the whole distributed application, which describes how the services and their backing resources fit together. In this system the second one is the Aspire orchestrator, and, while not necessarily required for it to work, it's very helpful and the more interesting of the two, so we'll start there.

The Orchestrator

One File That Describes the Whole System

Aspire lets you declare every service and every resource in plain C# or TypeScript, then runs the lot with one command.

The orchestrator project is the entry point of the system. It doesn't contain business logic. Its only job is to declare what exists and how it connects. You list the backing resources (key vault, storage, databases), you list the services (MCP server, agent container apps, the optional frontend), and you describe the references between them. Aspire takes that declaration and stands it up, wiring connection strings and service discovery so no service has to hardcode where another one lives.

The neat trick is that the same file behaves differently depending on whether you're running locally or publishing. Locally, the storage and database resources run as emulators in containers using Docker, so you need no cloud account to develop. When you publish, those same declarations resolve to real Azure resources. One description, two environments. It makes local development a breeze!

var builder = DistributedApplication.CreateBuilder(args);

// Backing resources. Locally these run as emulators in containers;
// at publish time the same handles resolve to real Azure resources.
var storage = builder.AddAzureStorage("storage").RunAsEmulator();
var blobs  = storage.AddBlobs("BlobStorage");     // the stage-to-stage bridge
var tables = storage.AddTables("TableStorage");   // finished deliverables

var cosmos = builder.AddAzureCosmosDB("cosmosdb").RunAsPreviewEmulator();
var memory = cosmos.AddCosmosDatabase("AgentMemory");
memory.AddContainer("ConversationSessions", "/deliverableId"); // run transcripts

// The tool server has to be ready before any agent can call it.
var tools = builder.AddProject<Projects.ToolServer>("toolserver");

// Stage one: intake. Reads/writes storage, talks to the tool server.
var intake = builder.AddProject<Projects.IntakeHost>("intakehost")
    .WithReference(blobs).WithReference(tables)
    .WithReference(tools).WaitFor(tools);

// Stage two: processing. Adds the conversation memory and a back-reference
// to intake so it can post progress updates across the process boundary.
var processing = builder.AddProject<Projects.ProcessorHost>("processorhost")
    .WithReference(blobs).WithReference(tables).WithReference(memory)
    .WithReference(tools).WaitFor(tools)
    .WithReference(intake);

builder.AddProject<Projects.OperatorUi>("operatorui")
    .WithReference(intake).WithReference(tools);

builder.Build().Run();

Read that top to bottom and you've read the system's wiring diagram. Two storage handles (one for the bridge, one for finished work), a Cosmos database for agent session persistence, the tool server, the two workflow container app projects, and the optional frontend. The WaitFor(tools) calls say "don't start the agents until the tools are up," and the WithReference calls are what give each service the connection details for the things it depends on. Notice the deliberate asymmetry: processing references intake (it posts progress back to it), but intake does not reference processing. Intake writes a work item and walks away. That one-way reference is the storage seam showing up in the wiring.

A general lesson, not an Aspire one

Even if you never touch Aspire, the concept remains the same: keep one declarative description of how your services and resources fit together, and make it environment-agnostic so local and production differ only in what the resource handles resolve to. Aspire is a tidy way to do this in .NET, but the value is in the single source of truth, not the specific API.

Inside a Service

Wiring One Process, From Config to Agents

Each service has its own small root that turns configuration and credentials into ready-to-run agents.

Drop down a level into any one of the services and you find the familiar .NET host builder. Each service calls AddServiceDefaults(), the shared Aspire helper that turns on health checks, OpenTelemetry, and resilient HTTP handlers in one line, and then registers the handful of things it needs. The most important of those is the credential chain, because it's how the keyless design from the prerequisites chapter becomes real.

var builder = WebApplication.CreateBuilder(args);
builder.AddServiceDefaults(); // Aspire: health checks, OpenTelemetry, resilience

// No API keys anywhere. One credential chain tries each source in order and
// uses the first that works: your CLI login locally, managed identity in Azure.
builder.Services.AddSingleton<TokenCredential>(new ChainedTokenCredential(
    new AzureCliCredential(),
    new EnvironmentCredential(),
    new ManagedIdentityCredential()));

// Everything the agents need is registered once, here, and injected elsewhere.
builder.Services.AddSingleton<IChatClientProvider, ChatClientProvider>();
builder.Services.AddSingleton<EntityStore>();
builder.Services.AddSingleton<AgentPool>();

var app = builder.Build();
app.MapDefaultEndpoints();
app.Run();

The piece worth lingering on is AgentPool. Rather than scatter agent construction across the codebase, the system funnels it through one component that knows how to build a configured agent. When something needs the researcher, it asks the pool; it never news up a ChatClientAgent itself. That centralization is what lets the middleware stack from Chapter 11 be applied uniformly: the pool builds the model client, wraps it in the fluent chain, and hands back a finished agent.

// Inside AgentPool: build the model client, wrap it in cross-cutting layers,
// then construct the agent. Every agent in the system is instantiated this way.
IChatClient client = chatClientProvider.GetChatClient()
    .WithRateLimitRetry()
    .WithAgent365Baggage(baggage)
    .WithPurview(purviewOptions)
    .WithOpenTelemetry()
    .WithCompaction(compactionProvider);

var researcher = new ChatClientAgent(client, new ChatClientAgentOptions
{
    Name = "ResearchAgent",
    Instructions = researchInstructions,
    ChatOptions = new ChatOptions { Tools = mcpTools }
});

Don't worry about what each .With* layer does yet; that's Chapter 11. The point here is structural: the agent is constructed in exactly one place, from injected dependencies, with its cross-cutting concerns layered on at birth. Everywhere else in the code, an agent is just something you were handed.

Why It Pays Off

Keeping the Root Thin and Everything Else Ignorant

A disciplined composition root is what makes every later chapter's "just swap this" claim actually true.

It's tempting to treat the composition root as boilerplate and stop reading. Resist that, because the discipline here is what pays off everywhere else in the guide. When the only place that constructs a model client is the agent pool, swapping Azure OpenAI for a hosted prompt agent (Foundry or another service) is a change in one method, not a hunt through the codebase. When the only place that declares storage is the orchestrator, moving from emulators to real Azure is a publish-time concern, not a code change. When credentials come from an injected TokenCredential, the agents never learn whether they're running on your laptop or in the cloud.

The rule of thumb is to construct at the edges, inject everywhere else. The root knows how things are built; nothing else does. Maintain that boundary and the system stays clean enough to reshape. Blur it (new up a client deep inside an agent, hardcode a connection string in a handler) and every future change gets a little harder.

With the wiring understood, we can start following the work. The first thing that runs after an event arrives is the intake stage, and it's built as an explicit two-node workflow graph. That's where we go next.
