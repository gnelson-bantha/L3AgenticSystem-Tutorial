Chapter 16

Deployment

Running the system on a laptop and in the cloud from one description, with Aspire keeping the two honest.

Parity by Construction

One Description, Two Environments

The orchestrator from Chapter 5 wasn't just a dev convenience. It's also the deployment model!

Back at the composition root we described the system's resources (storage, the database, the services, the references between them) in the Aspire AppHost, and treated it as the thing that boots everything for local development. Here's the other neat thing about Aspire... that same description is the deployment model. Aspire reads the AppHost in one of two modes. In run mode it wires everything up locally against emulators. In publish mode it provisions the real cloud resources and deploys the services to them. You don't maintain a separate stack of deployment scripts that slowly drifts away from how you actually run the thing day to day.

That's worth pausing on, because dev/prod drift is where a scary amount of operational pain comes from. When the local setup and the production setup are described by different artifacts, they diverge, and the divergence is exactly what you didn't test. Ever hear someone say, "but it worked on my machine?!!?" Yeah... I think we all have. Aspire's answer to this problem is to make them the same artifact read two ways.

On Your Machine

Running It Locally

Emulated cloud services in containers, one command, and the same credential flow you'll use in production.

Locally, the cloud dependencies run as emulators in containers, started for you by the AppHost. The storage account becomes Azurite, and the database becomes the Cosmos emulator, both spun up automatically. You don't install or configure anything by hand. You run the AppHost and the whole topology spins up services and their backing stores together, with the dashboard showing you every system.

// Run mode: the same resources come up as local emulators in containers.
var storage = builder.AddAzureStorage("storage").RunAsEmulator();          // Azurite
var cosmos  = builder.AddAzureCosmosDB("cosmos").RunAsPreviewEmulator();    // Cosmos emulator
# Start the entire system, emulators and all, from the AppHost:
aspire run

Two details to note from earlier chapters: First, the no-keys identity approach means that locally you're authenticated by your developer sign-in (the first link in the credential chain), with no secrets in a config file to leak or rotate. And second, because the services find each other by logical name through service discovery, the progress reporting from Chapter 13 works on your laptop exactly as it will in the cloud, no hardcoded localhost ports to adjust.

To the Cloud!

Publishing to Azure

The same AppHost, read in publish mode, provisions real services and ships the containers to them.

To run in the cloud you add the cloud-side targets to the same AppHost and let publish mode take over. A container app environment becomes the host for the services, the model backend points at a provisioned (or existing) Azure OpenAI resource, and an Application Insights resource gives the OpenTelemetry from Chapter 11 somewhere to land. The emulator calls are simply how those same resources behave in run mode. In publish mode they resolve to the genuine articles.

// Publish mode: the same model adds real cloud targets.
var env      = builder.AddAzureContainerAppEnvironment("env");
var openai   = builder.AddAzureOpenAI("openai");                 // provisioned, or .AsExisting(...)
var insights = builder.AddAzureApplicationInsights("insights");  // sink for the OTel traces

builder.AddProject<Projects.IntakeHost>("intakehost")
    .WithReference(openai)
    .WithReference(insights);
# Provision the resources from the AppHost model and deploy the services:
azd up

And the identity story closes itself out. In the cloud there's no developer sign-in, so the credential chain falls through to managed identity, which the deployment assigns to the services. The exact same code that authenticated against your local sign-in now authenticates as the workload's managed identity, with no API keys anywhere in the path, just as Chapter 3 promised. The one credential abstraction spans your laptop and production without a branch in the code.

Cloud Providers

While the sampled repository uses Azure services, it is important to note that Aspire is capable of being cloud-agnostic.

The Payoff

Why Same-Code Deployment Matters

Parity isn't a nicety here. It's what makes an autonomous system trustworthy to ship.

It's easy to read this as tooling convenience, but for an autonomous system the parity is closer to a safety property. The thing you exercised locally (the same services, the same identity flow, the same service discovery, the same telemetry wiring) is the thing you ship. There's no separate production-only configuration path that goes untested until the worst possible moment. Because the agents run without a human gate, you really do want the deployed system to behave like the one you debugged, and one description read two ways is how you get that.

What you tested ships

Local and cloud are the same AppHost in two modes, so production isn't an untested configuration you meet for the first time on deploy day.

One identity story

Developer sign-in locally, managed identity in the cloud, the same credential chain and zero keys in either place.

Observability on by default

The telemetry built into the middleware flows to the dashboard locally and to Application Insights in the cloud, with nothing to turn on.

That's the system, end to end: built from clean seams, run autonomously, kept observable and accountable, and shipped from a single description! The only thing left is to point you at the deeper material for every piece of the stack we leaned on, so you can build your own.
