Chapter 6

The Intake Stage: A Linear Workflow Graph

A small, fixed agent pipeline that turns one raw event into many concrete units of work.

Tool Choice

When the Job Is a Straight Line

Not every multi-agent task needs agents debating; sometimes you just need a linear flow

The composition root handed us configured agents. Now we put the first two to work. Intake has a job with a fixed shape: take the incoming event, clean it up, then figure out which targets it affects. In the source repository, the targets were retailers. Those two steps always happen in that order, every time. There's no judgment call about who should act next, no back-and-forth, no looping. It's a linear interaction, which doesn't necessarily have to be agentic, but serves in this pattern to demonstrate how two agentic workflows can work together with a data bridge.

That matters, because the Microsoft Agent Framework SDK gives you more than one way to coordinate agents, and picking the right one determines how your agents will be orchestrated. When agents need to collaborate and the path isn't fixed, you reach for a Group Chat where a manager decides who speaks (that's the processing stage, Chapter 9) or a Magentic workflow where the agents talk to each other in a self-directed manner. When the path is fixed, you reach for an explicit workflow graph: nodes wired with edges, messages flowing along them. Intake is the second case, so we build it with WorkflowBuilder and keep it simple.

A small rule worth following

Match the orchestration to the determinism of the task. A predictable sequence deserves a graph you can read at a glance; an open-ended collaboration deserves a manager that can adapt. Using a Group Chat for a straight line buries a simple thing in machinery, and using a fixed graph for an open-ended task fights the problem. Intake earns the simpler tool.

The Graph

Two Nodes and an Edge

Each step is an executor; an edge connects them; typed messages flow from one to the next.

In the Microsoft Agent Framework SDK, a node in a workflow graph is an executor: a unit that receives a typed message, does something, and optionally sends a message onward. You build one by deriving from ReflectingExecutor and declaring which message types it handles with IMessageHandler. The graph itself is just executors joined by edges. Here's the whole intake pipeline, assembled.

// Two derivatives of ReflectingExecutor, one edge. That is the entire intake workflow.
var normalizer = new NormalizerExecutor(agentPool);
var selector   = new SelectorExecutor(agentPool, bridge);

var workflow = new WorkflowBuilder(normalizer)   // normalizer is the entry node
    .AddEdge(normalizer, selector)               // its output flows to selector
    .Build();

// A run starts when the REST trigger hands in a freshly received event.
await workflow.RunAsync(new InboundEvent(correlationId, rawText));

The first executor normalizes. It takes the raw event, asks the normalizer agent to enrich it (pull out the key implications, tidy the structure), and then sends a typed WorkRequest down the edge. The thing to notice is context.SendMessageAsync. That's how an executor passes a result to whatever the graph wires up next. The executor doesn't know or care who receives it. The edge decides.

internal sealed class NormalizerExecutor
    : ReflectingExecutor<NormalizerExecutor>, IMessageHandler<InboundEvent>
{
    public async ValueTask HandleAsync(InboundEvent message, IWorkflowContext context)
    {
        var agent    = agentPool.GetNormalizer();
        var enriched = await agent.EnrichAsync(message.RawText);

        // Pass the enriched event to the next node. The edge decides who that is.
        await context.SendMessageAsync(
            new WorkRequest(message.CorrelationId, enriched));
    }
}

That's the entire mechanism. Typed messages in, typed messages out, edges deciding the flow. If you later needed a third step (say, a deduplication pass between normalize and select), you'd add an executor and an edge, and nothing else would change. The graph stays legible because it's literally a list of nodes and the edges between them.

Fan-Out

From One Event to Many Work Items

The second node is where a single event explodes into the pile of independent jobs the rest of the system feeds on.

The selector executor is where intake does the bulk of the work to prepare for fanning out to the next agentic workflow. It runs the selector agent, which has the tool-server query tools bound to it (those come from the next chapter), and asks, "of all the targets we track, which ones does this event actually affect, against every stated criterion?" The agent reasons over the data the tools return and produces a list of matches. Then the executor writes one work item per match into the storage bridge.

internal sealed class SelectorExecutor
    : ReflectingExecutor<SelectorExecutor>, IMessageHandler<WorkRequest>
{
    public async ValueTask HandleAsync(WorkRequest message, IWorkflowContext context)
    {
        // The selector agent queries target data over MCP and decides who matches.
        var selector = agentPool.GetSelector();
        var targets  = await selector.SelectTargetsAsync(message.Event);

        // Fan-out: one independent work item per matched target, keyed under the
        // correlation id so the whole event's output groups together in the bridge.
        foreach (var target in targets)
        {
            var item = new WorkItem(message.CorrelationId, target, message.Event);
            await bridge.WriteAsync($"{message.CorrelationId}/{target.Id}.json", item);
        }
    }
}

This is the fan-out that the whole parallel design depends on. One event in, many work items out. Each work item is self-contained. It carries the correlation id (required for Agent 365 observability), the target it's about, and the event context, so the processing stage can pick it up and run with zero knowledge of intake. The blob path encodes the correlation id as a prefix, which means every work item born from one event lands under the same folder, a small detail that makes recovery and tracking much easier later.

The selector doesn't call the processing stage, doesn't start any runs, and doesn't wait for anything. It writes blobs and finishes. The fan-out is expressed purely as data sitting in storage, waiting for the processing stage to pick it up. This is how these two agentic workflows are communicating without one blocking the other and allowing the system to scale horizontally.

The Handoff

Why Intake Stops at the Storage Edge

The most important thing about this stage is the line it refuses to cross.

It would be easy to make intake "helpful" by having it kick off processing for each work item it creates. Don't. The moment intake calls into processing, you've welded the two stages together and made scaling horizontally much more difficult. They now have to be deployed together, scaled together, and they fail together. Every reason we split the system in the first place is invalidated.

So intake stops at the storage edge on purpose. Its contract with the rest of the system is exactly one thing: work items appear in the bridge. What reads them, when, and how many at a time is none of its business. That restraint is what keeps intake cheap and fast — a fresh intake instance is spawned for each incoming event, does its short run, and exits — while processing stays free to scale as-needed.

Two lessons run out of this chapter. One is the tools the selector used to query target data, which deserve a close look because they live behind a protocol boundary on purpose. The other is the storage bridge those work items landed in. We'll take the tools first, since the selector just leaned on them, and come back to the bridge in Chapter 8.
