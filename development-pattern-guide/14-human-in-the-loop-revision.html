Chapter 14

Human-in-the-Loop Revision

Letting a person re-open a finished deliverable and re-run the very same agents, without softening the system's autonomy.

A Careful Distinction

Autonomy Doesn't Mean Unaccountable

There's a real difference between a human steering every step and a human reacting to a finished result, and it's the difference between Level 2 and Level 3.

It's tempting to think that letting humans intervene at all would knock the system back down to Level 2. It doesn't, and the reason is about when the human acts. Level 2 puts a person inside the execution loop: the agent does a step, waits for a human to approve, does the next step, waits again. The human is a gate the run cannot pass without. That's not what happens here. Here the agents run a complete job end to end with no human gate at all, produce a finished deliverable, and only then can a person look at it and say "actually, change this."

That after-the-fact feedback doesn't reach into a paused run; it kicks off a brand new autonomous run. The human supplies a direction and steps back out, and the agents go off and revise on their own, hitting the same review loop and the same termination rules as before. So the autonomy is intact: every run, original or revision, executes without a human in the middle of it. What we're adding is accountability, the ability to push back on a result, not supervision of each step. Keeping that line clear is what lets the system be genuinely autonomous and genuinely answerable to people at the same time.

Human feedback here doesn't pause an autonomous run. It starts another one.

Foresight

Why the Transcript Was Worth Saving

Revision is only cheap because the original run left behind everything you'd need to resume it.

Re-running the agents would be painful if you had to start cold, re-research everything, and hope the second attempt resembled the first. The system avoids that because every run persisted its full conversation to the session store (the Cosmos database wired up back at the composition root). That record holds the complete transcript of who said what, how many times the work has been revised, and the history of any feedback already applied.

That stored context is the quiet enabler of this whole chapter. When a revision comes in, you don't rebuild the world; you reload the run's memory and continue from it. The agents see the original brief, the research they already did, the drafts and critiques they already exchanged, and on top of all that, the new human request. The revision is an informed continuation rather than a fresh start, which is both cheaper and far more coherent.

The transcript

The full back-and-forth of the original Group Chat, so the agents resume with the same shared context they built the first time.

The revision count

How many times this deliverable has already been reworked, so you can reason about, and cap, how much human-driven churn one item absorbs.

The feedback history

Every prior human request, so a new revision builds on earlier direction instead of contradicting or forgetting it.

Reuse

Re-Entering the Same Loop

A revision doesn't get its own special machinery; it borrows the exact pipeline the original run used.

A person submits feedback by posting it to an endpoint keyed on the deliverable's id. That flips the deliverable into a "being modified" state and queues the revision. The handler that does the work is where the elegance lands: it loads the saved session, builds the same Group Chat from Chapter 9, and resumes the conversation with the human's note added as the newest message. Everything downstream, the round-robin, the reviewer's approval gate, the bounded loop, the middleware stack, behaves exactly as it did on the first run, because it literally is the same code.

app.MapPost("/api/deliverables/{id}/feedback", async (
    string id, HumanFeedback feedback, DeliverableService svc) =>
{
    await svc.BeginRevisionAsync(id, feedback);  // sets status to Modifying, queues work
    return Results.Accepted();
});
public async Task RunRevisionAsync(string deliverableId, HumanFeedback feedback)
{
    // Resume, don't restart: reload the original run's memory from the session store.
    var session = await sessionStore.LoadAsync(deliverableId);
    session.RevisionCount++;
    session.FeedbackHistory.Add(feedback);

    // The very same Group Chat as the first run, seeded with the prior transcript
    // plus the human's request as the latest message. Same agents, same loop, same cap.
    var workflow = BuildGroupChat();
    var result   = await workflow.ResumeAsync(session.Transcript, feedback.AsMessage());

    // Update in place: the deliverable keeps its id, its correlation id, and its history.
    await deliverableStore.UpdateAsync(deliverableId, result.FinalText, status: Reviewing);
    await sessionStore.SaveAsync(session);
}

Notice the update is in place. The deliverable keeps its identity (the same id, the same correlation id that's threaded everywhere from the last chapter, the same lineage) rather than spawning a new record. A revision is a new version of one thing, not a second thing, and the storage reflects that. This is the composition discipline from Chapter 5 paying a dividend you might not have predicted: because the Group Chat was built from injected, reusable parts, the revision path got to reuse it wholesale instead of reimplementing it.

State Machine

A Lifecycle That Allows Return

None of this works unless the deliverable's states were designed to let finished work flow backward.

The thing that makes revision possible at the data level is that the deliverable's status was modeled as a small state machine with a way back, not a one-way march to "done." A deliverable starts pending, moves to reviewing while the agents work, and settles at filed when it's approved. The addition for human feedback is a modifying state that any reviewable or filed deliverable can drop into, and from which it returns to reviewing once the agents finish their revision pass.

  Pending ──▶ Reviewing ──▶ Filed
                  ▲                │
                  │                │  human requests changes
                  │                ▼
                  └────────── Modifying
            (revision pass finishes, re-enters Reviewing)

The design lesson is to draw your state machine with the return paths in it from the start. It's easy to model only the forward happy path (created, working, done) and discover later that you've painted "done" as a terminal state with no way to reopen it. By giving the lifecycle an explicit modifying state and a documented route back to reviewing, the system makes human-driven iteration a first-class transition rather than a hack bolted onto a finished record. Work can come back, on purpose, through a door that was built for it.

That closes the loop on the pattern itself. We've gone from a raw event all the way to a reviewed, revisable deliverable, every piece of it autonomous where it counts and accountable where it should be. The remaining chapters are about living with the system: where to extend it, how to ship it, and where to read more. Extending comes first.
