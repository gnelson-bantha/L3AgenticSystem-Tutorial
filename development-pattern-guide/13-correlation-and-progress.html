Chapter 13

Correlation and Progress Across Hosts

One id that threads the whole pipeline, and progress that crosses the boundary between two separate processes.

The Visibility Problem

Following One Event Through Two Processes

The data bridge that bought us decoupling also took away the easy answer to "where is this event right now?"

The decoupling we worked so hard for has a flip side. Intake and processing are separate processes that never call each other, which is wonderful for scaling and terrible for answering a simple operator question: "I submitted an event a minute ago, what's happening to it?" There's no single call stack to follow, no one process that sees the whole journey. The event fans out into work items, those scatter across parallel runs in a different service, and nobody is holding the thread.

So the system holds the thread deliberately, with two complementary moves. First, a single correlation id is born the instant an event arrives and rides along on every artifact the event ever produces, so all of its scattered activity can be stitched back together after the fact. Second, the processing service actively reports progress back across the boundary, so the operator can watch in something close to real time. The id is for correlation; the reporting is for liveness. Together they give you a coherent view of a deliberately incoherent topology.

The Golden Thread

One Id, Stamped on Everything

Born at the front door, the correlation id ends up on the blob path, the work item, the telemetry, and the final deliverable.

When the event lands, intake mints one identifier and refuses to let go of it. It travels into the normalized event, becomes the prefix of every work-item blob path, rides inside each work item as its run id, tags every progress update, and lands on the finished deliverable. Trace any artifact in the system back and the same id is on it, which means "show me everything that came from that one event" is always answerable.

The correlation id is minted once and stamped on every artifact downstream:

  intake     InboundEvent(correlationId)  ->  WorkRequest(correlationId)
  bridge     blob path:  {correlationId}/{targetId}.json
  work item  WorkItem.WorkflowRunId == correlationId
  progress   ProgressUpdate.CorrelationId
  result     Deliverable.CorrelationId
  telemetry  Agent 365 SessionId == correlationId   (the baggage from Chapter 11)

That last line closes a loop from two chapters ago. The Agent 365 baggage built into the middleware stack uses this exact correlation id as its session id. So the identity telemetry emitted deep inside a model call, in a different process, is keyed to the same value that's on the blob path back in intake. You can pivot from a deliverable to its telemetry to its work items without ever guessing which records belong together. One worthwhile detail: as the Agent 365 SDK reached general availability it dropped its separate correlation field, so the session id is the carrier now. Folding correlation into the session id isn't a workaround; it's the supported way to group an event's activity.

Mint early, thread relentlessly

The discipline that makes this work is boring and absolute: create the id at the very first point of entry, put it on every record and message that follows, and reuse it as the key for logs, metrics, and traces. It costs almost nothing at the start and is nearly impossible to retrofit later. In a fan-out system with no shared call stack, this single thread is often the only thing standing between you and an unobservable mess.

Phoning Home

Reporting Progress Back Across the Boundary

Processing has no screen of its own, so it tells intake what's happening over plain HTTP.

The processing service runs headless in the background; it has no UI. The operator UI, meanwhile, lives next to the intake host. So processing reports progress to intake, and intake becomes the single place that knows the current state of every event. The mechanism is refreshingly ordinary: processing makes HTTP calls to a couple of endpoints on the intake host, one for incremental progress and one to mark a target complete.

public sealed class ProgressReporter(HttpClient intake)
{
    // Incremental update as a run moves through its stages.
    public Task ReportAsync(ProgressUpdate update) =>
        intake.PostAsJsonAsync("/api/workflow/progress", update);

    // Fired when one target's deliverable is finished (or finished-as-draft).
    public Task CompleteAsync(string correlationId, string targetId) =>
        intake.PostAsJsonAsync("/api/workflow/complete",
            new CompletionUpdate(correlationId, targetId));
}

The detail that keeps this clean is how processing finds intake. It doesn't hardcode a URL. The HTTP client is configured against the intake service's logical name, and Aspire's service discovery resolves that name to the right address in whatever environment you're running, your laptop or the cloud. This is the back-reference from the orchestrator chapter showing up in code: the wiring said processing references intake, and here's what that reference is for.

// No hardcoded address. Aspire resolves the logical service name "intakehost"
// to the real endpoint, the same locally and in production.
builder.Services.AddHttpClient<ProgressReporter>(c =>
    c.BaseAddress = new Uri("https+http://intakehost"));
The Window In

The Operator's View

With intake aggregating everything by correlation id, the UI just has to ask and render.

Because intake collects every progress and completion report, it can answer "what's the state of everything?" in one call. The operator UI polls a status endpoint and gets back a list keyed by correlation id, with the detected, in-flight, and completed counts for each event, drawn from the same thread-safe tally the bridge maintained back in Chapter 8. The UI gives each event its own tab and shows the familiar "this many of that many done," ticking up as completion reports arrive.

// Intake has aggregated everything by correlation id; the UI just polls and renders.
var events = await http.GetFromJsonAsync<IReadOnlyList<EventStatus>>(
    "/api/workflow/status");

// EventStatus per event: correlation id, detected / inFlight / completed counts,
// and the per-target state behind each event's tab.

It's worth noticing how little the UI has to know. It doesn't talk to processing, doesn't read the bridge, doesn't understand the agents. It asks intake one question and draws the answer. All the cross-process complexity was absorbed by the correlation id and the progress reports, leaving the front end with an almost trivial job. That's the payoff of doing the threading properly: visibility becomes a rendering problem instead of a distributed-systems problem.

This progress plumbing also quietly sets up the final piece of the pattern. The status the operator sees includes deliverables that finished, and a finished deliverable is something a human might want to push back on. Re-opening one and sending it back through the same agents is the last behavior we'll build, and it's next.
