Appendix: Additional Reading

The primary sources for every piece of the stack this pattern leaned on, so you can go build your own.

How to Use This

Going Deeper on the Stack

The pattern is portable, but the implementation we studied used a specific set of Microsoft technologies. Here's where to learn each one properly.

This guide deliberately stayed at the level of the pattern, leaning on the underlying technologies just enough to make each idea concrete. When you go to build something like it, you'll want the real documentation for the pieces you choose. The list below is grouped roughly the way the chapters were: the agent framework and its orchestration first, then the tool protocol, then the governance and observability layers, then the Azure services that act as the durable seams, and finally the deployment tooling. Every link points to official, primary material for the corresponding part of the stack.

Microsoft Agent Framework (Documentation)

The framework behind the agents, the workflow graph from the intake stage, and the Group Chat orchestration with its manager and termination rules. Start here for chat clients, agents, workflows, and the multi-agent patterns the collaboration stage is built on.

https://learn.microsoft.com/en-us/agent-framework/

Microsoft Agent Framework (Source and Samples)

The open-source repository, including the workflow and Group Chat APIs, the delegating chat-client middleware surface, and the experimental context-compaction strategies covered in the compaction chapter. Read the samples alongside the docs.

https://github.com/microsoft/agent-framework

Aspire

The orchestration and composition model that defines the system's resources once and runs them against local emulators or real cloud services. Covers the AppHost, service discovery, resource references, and run-versus-publish deployment, the backbone of the composition-root and deployment chapters.

https://learn.microsoft.com/en-us/dotnet/aspire/

Model Context Protocol (Specification)

The open protocol that puts tools behind a process boundary, as in the tool-access chapter. The specification defines tool discovery, invocation, and transports, and explains why an agent and its tools can evolve and scale independently.

https://modelcontextprotocol.io

Model Context Protocol C# SDK

The .NET implementation used to build the tool server and connect clients to it, including the server-tool attributes, HTTP transport, and the client-side tool abstractions the agents call through.

https://github.com/modelcontextprotocol/csharp-sdk

Azure AI Foundry

The platform for hosted agents and model deployments. Relevant when you swap an agent's backend for a Foundry-defined agent reached through the project client, as discussed in the extending chapter, and for managing the models the agents run against.

https://learn.microsoft.com/en-us/azure/ai-foundry/

Azure OpenAI

The model backend the agents call, wired up as a referenced resource in the AppHost. Covers deployments, the chat completions surface the chat client sits on top of, and the keyless authentication path the system relies on.

https://learn.microsoft.com/en-us/azure/ai-services/openai/

Identity and Governance for Agents (Microsoft Purview and Agent 365)

The governance and identity layers applied as middleware: Microsoft Purview for data governance over agent interactions, and the Agent 365 baggage that stamps tenant, session, and agent identity onto each call. Purview's documentation is the entry point for the governance side.

https://learn.microsoft.com/en-us/purview/

OpenTelemetry in .NET

The observability standard the telemetry middleware emits into, and which the deployment sends to Application Insights. Covers traces, metrics, and logs in .NET, the foundation for following an autonomous run you didn't watch happen.

https://learn.microsoft.com/en-us/dotnet/core/diagnostics/observability-with-otel

Azure Cosmos DB

The session store that persists each run's transcript, revision count, and feedback history, the thing that makes human-in-the-loop revision a resume rather than a restart. Covers containers, partition keys, and the local emulator used in development.

https://learn.microsoft.com/en-us/azure/cosmos-db/

Azure Storage (Blobs and Tables)

The durable storage bridge between the intake and processing stages, where work items are written, claimed, and processed in parallel. Covers Blob and Table storage and the Azurite emulator that stands in for them locally.

https://learn.microsoft.com/en-us/azure/storage/

Keyless Authentication with Azure Identity

The credential model behind the no-keys posture: a chained credential that resolves to your developer sign-in locally and to a managed identity in the cloud, with the same code in both places. The reference for the deployment chapter's identity story.

https://learn.microsoft.com/en-us/dotnet/azure/sdk/authentication/

Azure Container Apps and the Azure Developer CLI

The deployment target and tooling for publish mode. Container Apps hosts the services, and the Azure Developer CLI provisions the resources described in the AppHost and ships the containers, turning one description into a running cloud system.

https://learn.microsoft.com/en-us/azure/container-apps/
Last Word

Now Go Build One

The pattern matters more than any single library on this page.

If you take nothing else from this guide, take the shape: an autonomous, multi-stage agent pipeline whose stages are decoupled by durable storage, fanned out for parallelism, self-corrected by a bounded generate-and-review loop, and wrapped in composable middleware for everything cross-cutting. Every technology on this page is replaceable. The seams between them are the design. Build along those seams and your own system will be observable, extensible, and honest about what it does on its own, which is the whole point of working at Level 3.
