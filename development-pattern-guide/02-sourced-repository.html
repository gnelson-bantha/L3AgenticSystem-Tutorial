Ground Truth

What the Project Is

Before we abstract anything, here's the concrete thing on disk, named the way its authors named it.

Every chapter after this one uses generalized names so the lessons travel. This chapter is the deliberate exception. The pattern was lifted from a Microsoft sample called the Sample Level 3 Agentic System, and to understand what was studied you need to see it as it really is, real project names and all. So for these few pages, the gloves come off and we use the actual identifiers.

The sample's use case is Retailer Impact Evaluation. A "change event" arrives (some regulatory shift, a supply disruption, a market move), and the system autonomously works out which retailers it affects, then generates a per-retailer impact report through an agent feedback loop, and delivers the finalized reports with no human intervention during execution. It's a proof-of-concept by design: it uses blob polling as an event bus and simplified data models, but the architecture is built so that production swaps (Event Grid, retrieval pipelines, dynamic agent registration) drop in without restructuring.

The stack is .NET 10 throughout, orchestrated by Aspire. The agents are built on the Microsoft Agent Framework SDK (offering C# and python support), talking to Azure OpenAI through ChainedTokenCredential so there are no API keys anywhere, just RBAC. Data lives in Azure Table Storage and Azure Cosmos DB, the stage-to-stage bridge is Azure Blob Storage, the UI is Blazor Server, and tool access runs over the Model Context Protocol. Optional integrations layer in Agent 365 observability, Microsoft Purview governance, Azure AI Foundry prompt agents, Microsoft Fabric data agents, and Logic Apps notifications.

Topology

Two Hosts Joined by a Blob Container

The solution splits at one clean line, and that line is a storage account.

The two halves of the system are two ASP.NET Core projects. ChangeEventEvaluatorHost is the intake workflow: it exposes the REST trigger, runs the first two agents, and writes one blob per matched retailer to the retailer-work-items container. ChangeEventProcessorHost is the processing workflow: a background service polls that container every five seconds, and each new blob launches an independent per-retailer run.

Why two projects instead of one? Independent scaling. Intake runs once per change event; processing runs once per affected retailer and is far more expensive. Splitting them at Azure Blob Storage means you can run many ChangeEventProcessorHost instances chewing through retailers in parallel while a single ChangeEventEvaluatorHost handles intake. The blob container is the entire contract between them, which is exactly why the bridge is so flexible.

The two workflows even use different orchestration styles from the Agent Framework, on purpose. Intake is a fixed two-node graph built with WorkflowBuilder. Processing is a Group Chat built with AgentWorkflowBuilder.CreateGroupChatBuilderWith(). We'll dig into both later; for now just notice that the codebase reaches for the simpler tool when the job is simple and the richer one when agents need to collaborate.

SampleL3AgenticSystem/
├── AppHost/                       Aspire orchestrator — wires every service and resource
├── ServiceDefaults/               Shared Aspire infra: health checks, OpenTelemetry, resilience
├── ChangeEventEvaluatorHost/      Intake workflow — REST API + Observer/Evaluation agents
├── ChangeEventProcessorHost/      Processing workflow — Researcher/Generator/Reviewer Group Chat
├── SampleMCPServer/               MCP server exposing retailer-data query tools
├── DemoUI/                        Blazor Server frontend for triggering and monitoring
├── SampleL3AgenticSystem.Shared/  Models, abstractions, storage services, DTOs
└── M365DeclarativeAgent/          Microsoft 365 Copilot declarative agent (no code, API plugin)
The Cast

The Five Agents

Two live in intake and run as workflow executors; three live in processing and collaborate in a Group Chat.

The intake host runs two agents as ReflectingExecutor nodes wired together with WorkflowBuilder. ChangeEventObserverAgent normalizes the raw event and pulls out its key implications. ChangeEventEvaluationAgent then queries retailer data through MCP tools, decides which retailers match every stated criterion, and writes a RetailerWorkItem blob for each one. These two pass typed messages to each other and route results with context.SendMessageAsync().

The processing host runs three agents as ChatClientAgent instances inside a Group Chat managed round-robin by a custom ChangeEventGroupChatManager. Researcher gathers economic-impact research for one retailer. ReportGenerator turns that research into a Markdown report. ReportReviewer is the quality gate: it answers either APPROVED, which ends the chat, or with revision feedback, which sends the team around for another pass. All three talk in natural language in the shared conversation, so there's no typed routing between them at all.

ChangeEventObserverAgent

Intake, executor node. Normalizes the raw ChangeEventTrigger into an enriched ChangeEvent with key implications. No tools.

ChangeEventEvaluationAgent

Intake, executor node. Uses MCP query tools to match retailers against every criterion, then writes one RetailerWorkItem blob per match.

Researcher

Processing, Group Chat. Gathers impact research via MCP tools. Optionally runs as an Azure AI Foundry prompt agent instead of direct Azure OpenAI.

ReportGenerator

Processing, Group Chat. Writes and revises the Markdown impact report from the research in the shared conversation.

ReportReviewer

Processing, Group Chat. The quality gate. Replies APPROVED to end the chat, or gives revision feedback to continue the loop.

The non-agents

BlobTriggerService, ProgressReporter, and SaveReportAsync() are plain infrastructure: polling, cross-process status, and persistence. No LLM calls.

Supporting Code

The Shared Library and the Edges

A lot of the interesting machinery sits outside the agents, in code both hosts lean on.

SampleL3AgenticSystem.Shared is where the reusable pieces live. It holds the domain records in Models.cs (ChangeEvent, RetailerProfile, RetailerWorkItem, RetailerReport, and friends), the IChatClientProvider abstraction over the model client, the TableStorageService, and the file that does a lot of quiet heavy lifting: ChatClientPipelineExtensions. That last one defines the fluent IChatClient wrappers (.WithRateLimitRetry, .WithAgent365Baggage, .WithPurview, .WithOpenTelemetry, .WithCompaction) that both hosts use to assemble their agents. Chapter 11 is essentially a close reading of that one file.

SampleMCPServer is a standalone MCP server, built with ModelContextProtocol.AspNetCore and Streamable HTTP transport, that exposes the retailer-data query tools plus the workflow-control tools the declarative agent uses. ServiceDefaults is the standard Aspire shared-infra project (health checks, OpenTelemetry wiring, resilience handlers). AppHost is the Aspire orchestrator that ties everything together, which is where the next-to-last detail lives.

On the edges, DemoUI is a Blazor Server app for submitting mock events and watching the agents work, and M365DeclarativeAgent is a code-free Microsoft 365 Copilot agent that routes through SampleMCPServer to submit events and check status. Both are just triggers; the system could as easily be driven by a service that continuously scans for real events.

The Bridge to the Pattern

What the Rest of the Guide Generalizes

Now that you've seen the real thing, here's the translation key for everything that follows.

From here on, the retailer-specific names step aside so the pattern can stand on its own. The two hosts become the Intake Service and the Processing Service. ChangeEvent becomes an inbound event, RetailerProfile becomes a target entity, RetailerWorkItem becomes a work item, and the report becomes a deliverable. The five agents keep their roles but lose the domain flavor: a normalizer and a selector in intake; a researcher, a generator, and a reviewer in processing.

What does not get generalized is the Microsoft stack, and that's a deliberate call. The Agent Framework, MCP, Aspire, Agent 365, Purview, Azure AI Foundry, OpenTelemetry: those integrations are part of the pattern, not incidental detail. When Chapter 11 talks about wrapping an agent's client for Agent 365 identity telemetry, it names Agent365BaggageChatClient and the .WithAgent365Baggage extension, because that's the actual technique worth learning. Generalizing those away would leave you with a vaguer, less useful guide.

With the real codebase in view and the translation key in hand, the next chapter covers what you'll want installed and understood before building your own version.
