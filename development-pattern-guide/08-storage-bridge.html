Chapter 8

The Storage Bridge for Parallel Scale

Why a plain blob container sits between the two halves of the system, and how it turns one event into many parallel runs.

The Bridge

The Most Important Box Is Almost Empty

It's just a container of files, and that simplicity is what makes it clever.

The bridge between intake and processing is a blob container. This is the simplest possible way to enable communication between the two agentic workflows. It could be swapped for a database, a service bus, or any other messaging system. Intake drops a work item into it while processing fishes work items out of it. There's no direct call between the two services. Two workflows, in separate app containers, one folder of files, completely ignorant of each other.

It's worth being precise about what this storage container actually buys, because every benefit traces back to one property: the work item is durable the instant it's written. Once a work item is a blob, intake's job is genuinely over, and the work persists. Restart processing, deploy a new version, crash a node, and the work items are still sitting there waiting. The container is also a natural backpressure valve. If work arrives faster than processing can chew through it, items just accumulate, no one falls over, and the backlog drains when capacity frees up. Also, because the contract is "a work item is a blob," you can run as many processing instances as you like against the same container.

Put durable storage on the bridge between your stages, and a crash stops being a catastrophe. It's merely a speedbump.

Pickup

Polling as a Stand-In for Events

A background service watches the container and turns each new file into a running pipeline.

Something has to notice new work items and start runs. In this system that's a background service that polls the container on a short interval. Polling is a deliberately simple stand-in for a real event subscription; we'll talk about the production version at the end of the chapter. What it does on each pass is the interesting part: for every pending item, it claims the item by moving the blob into a separate processing container, then launches a run without waiting for it to finish.

Those two moves are doing a lot. The claim-by-move is how two processing instances avoid grabbing the same work item: once a blob has moved out of the pending container, no one else sees it as pending. And launching without awaiting is what makes the system parallel. The poller fires off run after run, each on its own, so ten work items become ten pipelines running side by side instead of a queue of ten waiting their turn.

public sealed class WorkQueuePoller(
    BlobBridge bridge, CollaborationRunner runner,
    AgentPool pool, EventTracker tracker) : BackgroundService
{
    protected override async Task ExecuteAsync(CancellationToken stopping)
    {
        await pool.WaitUntilReadyAsync(stopping);  // never poll before agents exist
        await bridge.RecoverOrphansAsync();        // re-claim anything a crash stranded

        while (!stopping.IsCancellationRequested)
        {
            await foreach (var name in bridge.ListPendingAsync(stopping))
            {
                // The claim IS the move: items in the processing container are no
                // longer visible to any other instance polling for pending work.
                var item = await bridge.ClaimAsync(name);
                if (item is null) continue;
                tracker.MarkInFlight(item.CorrelationId);

                // Fire and forget: do NOT await. This is where parallelism happens.
                _ = Task.Run(async () =>
                {
                    try     { await runner.RunAsync(item, stopping); }
                    finally { tracker.MarkCompleted(item.CorrelationId); }
                }, stopping);
            }

            await Task.Delay(TimeSpan.FromSeconds(5), stopping);
        }
    }
}

Two guardrails bracket the loop. The poller waits until the agent pool is ready before it polls at all, so a slow startup never causes a work item to be claimed and then dropped because the agents weren't built yet. And it recovers orphans before the first poll, which we'll cover next!

Resilience

Surviving Crashes With Orphan Recovery

Claiming an item by moving it avoids lost work items and duplicate processing.

Think about what happens if a processing instance dies mid-run. The work item it was handling has already moved into the processing container (it got claimed), but the run never finished. If nothing intervened, that item would sit in the processing container forever, claimed by a process that no longer exists - orphaned, forever alone, forgotten.

The claim-by-move pattern makes the fix almost free. Anything sitting in the processing container at startup must be an orphan, because a healthy instance would have finished it and removed it. So on startup, before polling, the poller sweeps the processing container and moves everything back to pending. Those items get picked up and run again. A crash costs you a restart, not a lost work item.

// Anything still in the processing container at startup was claimed by an instance
// that died before finishing. Move it back to pending so it gets picked up again.
public async Task RecoverOrphansAsync()
{
    await foreach (var orphan in ListProcessingAsync())
    {
        await MoveAsync(orphan, from: ProcessingContainer, to: PendingContainer);
    }
}

The other thing worth a look is how the system counts what's happening across all those parallel runs. Because many runs touch the counters at once, the tally has to be thread-safe, so it uses atomic increments rather than locks. A small EventTracker keeps, per event, how many work items were detected, how many are in flight, and how many have completed. That's what lets the UI later say "7 of 10 done" without any of the parallel runs tripping over each other.

public sealed class EventTracker
{
    private int _detected, _inFlight, _completed;

    public void MarkDetected()        => Interlocked.Increment(ref _detected);
    public void MarkInFlight(string id) => Interlocked.Increment(ref _inFlight);

    public void MarkCompleted(string id)
    {
        Interlocked.Decrement(ref _inFlight);   // atomic: safe under heavy parallelism
        Interlocked.Increment(ref _completed);
    }
}
Honest Trade-offs

What You'd Change for Production

The data bridge is the real design; polling a blob container is just the cheapest, easiest thing that honors it.

It's worth being upfront that polling a container every few seconds is not how you'd run this at serious scale. It adds latency (a work item can sit for up to the poll interval before anyone notices it) and it's chattier than it needs to be. The sample uses it because it's the simplest thing that proves the architecture.

The reason this is fine is that the data bridge, not the polling, is the design. The contract is "work items live in durable storage, and something picks them up." Swap the blob poll for a real event subscription (a storage-event trigger, a service bus queue, a managed event grid) and nothing on either side of the bridge has to change. Intake still writes durable work items; processing still claims and runs them. You'd keep the claim-by-move and orphan-recovery ideas, because durable queues have their own version of the same concerns. The poll is an implementation detail behind a deliberately stable interface.

The portable takeaway

Design the bridge as "durable work plus a pickup mechanism," then pick the cheapest pickup that fits where you are. Start with polling to get the architecture right, and graduate to a real event source without touching either stage. The decoupling is what's valuable; the transport is swappable.

We've now followed a work item from the selector's pen all the way to a running pipeline on the other side of the bridge. The obvious next question is what actually happens inside one of those runs. The answer is three agents in a shared conversation, and that's the next chapter...
