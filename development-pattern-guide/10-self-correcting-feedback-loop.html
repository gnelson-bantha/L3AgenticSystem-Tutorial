Chapter 10

The Self-Correcting Feedback Loop

A reviewer that can send work back, and a hard cap that guarantees the loop always ends.

The Core Problem

Quality Without a Human Gate

This is the chapter that earns the word "autonomous," because it's how output gets trusted with nobody watching.

Back in the introduction we raised the uncomfortable question at the center of Level 3 autonomy: if no human checks the work, what stops the system from confidently shipping something wrong? Every other chapter has been infrastructure leading to this one. Here's the answer the system gives: put the check inside the loop. Instead of a person reviewing the output, a peer agent reviews it, and it can send the work back for another pass.

The rhythm is generate, review, revise, repeat. The generator drafts the deliverable. The reviewer reads it against the brief and either approves it or returns specific, actionable feedback. If it's feedback, the team goes around again: the generator revises in light of the critique, the reviewer takes another look. The loop continues until the reviewer is satisfied. Because the critique is concrete and the generator can see the full conversation, each lap tends to be genuinely better than the last. That's what "self-correcting" means here, and it's how you get trustworthy output without a human in the execution path.

Termination, Part One

Teaching the Manager to Stop on Approval

The loop ends the moment the reviewer says the magic word, and the manager is what listens for it.

Recall from the last chapter that the Group Chat manager decides when the conversation is done. The default round-robin manager stops after a set number of turns, which is the wrong signal: we want to stop when the work is approved, not when the clock runs out. So the system subclasses the round-robin manager and overrides its termination check to watch for the reviewer's approval marker.

The reviewer is instructed to begin its message with one exact word, APPROVED, when and only when the deliverable is good. The manager checks the most recent reviewer message for that word. If it's there, the chat terminates and the deliverable is final. If it's not, the reviewer's message is treated as revision feedback and the loop continues.

public sealed class ApprovalGroupChatManager : RoundRobinGroupChatManager
{
    private const string ApprovalToken = "APPROVED";

    public ApprovalGroupChatManager(int maxIterations, int agentCount = 3)
    {
        // The hard ceiling, derived below. Belt to the approval check's braces.
        MaximumIterationCount = (maxIterations + 1) * agentCount;
    }

    public override ValueTask<bool> ShouldTerminateAsync(
        GroupChatThread thread, CancellationToken ct)
    {
        // Find the reviewer's latest verdict in the shared transcript.
        var verdict = thread.Messages
            .LastOrDefault(m => m.AuthorName == "ReviewerAgent")?.Text?.TrimStart();

        // Approval is the happy path: stop immediately.
        if (verdict is not null &&
            verdict.StartsWith(ApprovalToken, StringComparison.OrdinalIgnoreCase))
        {
            return ValueTask.FromResult(true);
        }

        // No approval yet: defer to the round-robin cap so the loop can't run forever.
        return base.ShouldTerminateAsync(thread, ct);
    }
}

The defensive details matter. The check trims leading whitespace and compares case-insensitively, because you don't want a stray space or a lowercase "approved" to prevent the system from ever recognizing success. The marker is a distinctive single word so it's unlikely to appear by accident in the middle of normal feedback. This is the careful handling of natural-language signals that the previous chapter promised.

Termination, Part Two

Bounding the Loop So It Always Ends

Approval is the way out you hope for; the cap is the way out you guarantee.

A reviewer that approves is the happy path. But what about a reviewer that's never quite satisfied? Maybe the task is genuinely hard, maybe the brief is contradictory, maybe the model just keeps finding nits. Without a backstop, the generate-and-review loop could run forever, burning tokens and never finishing. An autonomous system absolutely cannot have an unbounded loop, so the manager keeps a hard ceiling on total turns regardless of approval.

The ceiling is computed, not guessed. The idea is to allow a fixed number of revision iterations plus one initial pass, and since every iteration is one full round of the whole team, you multiply by the number of participants. With a default of three revisions and three agents, that's four full passes of three speakers, twelve turns, after which the manager stops no matter what.

// (maxIterations + 1) accounts for the initial pass plus the allowed revisions.
// Multiplying by the team size converts "iterations" into "speaker turns", since
// one iteration is one full round-robin lap of every participant.
MaximumIterationCount = (maxIterations + 1) * agentCount;
//   default: (3 + 1) * 3 = 12 turns, then the loop stops unconditionally.

Why a count, not a timeout

A turn count is deterministic and easy to reason about. You can say exactly how many model calls a run can cost in the worst case, which matters when thousands of runs go in parallel.

Why tie it to iterations

Expressing the cap as "allowed revisions" keeps the knob meaningful. You tune how many improvement passes you'll pay for, and the turn math follows from the team size.

Why bound at all

Autonomy plus an unbounded loop is a runaway bill and a hung pipeline waiting to happen. The cap is the single line that makes "runs on its own" safe to actually deploy.

Honesty

Failing Honestly When the Cap Wins

Hitting the ceiling without approval is a real outcome, and the system refuses to dress it up as success.

So the loop hit twelve turns and the reviewer still hasn't approved. What now? The wrong move is to quietly save the last draft as if it were finished and fire the same "all done" notification. That would launder an unfinished, unapproved deliverable into the record and tell everyone downstream it's good. In an autonomous system, that kind of dishonest success is exactly the failure mode you're trying to avoid.

So the system marks the outcome for what it is. The deliverable gets a distinct status that means "draft, not approved," a recorded reason explaining the loop ran out of revisions, and the reviewer's outstanding feedback appended right onto the document so a human can see precisely what was still wrong. Crucially, the success notification is suppressed: nobody gets told this one is finished, because it isn't.

var approved = result.WasApproved;   // did the reviewer sign off before the cap?

var deliverable = new Deliverable(item, result.FinalText);

if (!approved)
{
    // Tell the truth: this is a draft that ran out of revision budget.
    deliverable.Status       = DeliverableStatus.DraftIncomplete;
    deliverable.StatusReason = "Reached the revision cap without reviewer approval.";
    deliverable.Body        += "\n\n## Outstanding Reviewer Feedback\n"
                             + result.LastReviewerFeedback;
}

// Persist either way, but only announce completion when it's genuinely complete.
await SaveDeliverableAsync(deliverable, notifyOnSuccess: approved);
The principle worth stealing

An autonomous system will hit its limits sometimes, and the measure of a good one is what it does at the limit. Surface the incomplete state, attach the evidence of what's wrong, and stay silent on the success channel. A pipeline that fails loudly and honestly is one you can trust to run unattended; a pipeline that hides its failures is not.

That completes the engine room of the pattern: decoupled stages, parallel fan-out, and now a bounded, self-correcting loop that knows when to stop and how to admit when it couldn't finish. What's left is the cross-cutting machinery that wraps around all of it. Every model call these agents just made passed through a stack of middleware for retry, telemetry, governance, and more, and how that stack is built is one of the most reusable ideas in the whole system. That's next.
