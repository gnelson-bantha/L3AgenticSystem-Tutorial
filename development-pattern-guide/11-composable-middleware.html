Chapter 11

Composable Middleware for the Agent Client

Stacking retry, Agent 365 identity telemetry, Microsoft Purview governance, OpenTelemetry, and compaction as layers around each agent's model client.

The Idea

One Bridge, Many Concerns

How to offer support for services and features without cluttering the agents and keeping maintenance simple.

The agents you've watched do their work have stayed remarkably free of clutter. The researcher researches, the generator writes, the reviewer reviews. None of them contain a line about retrying a throttled call, emitting a trace, checking a governance policy, or trimming their own history. That's not an accident, and it's not because those concerns were skipped. They're all present - they just live somewhere else.

The "trick" is to use a single interface and extend its functionality with middleware. IChatClient is the interface every agent uses to talk to the model. Because every model call flows through that one interface, you can wrap it. A wrapper implements IChatClient too, holds a reference to an inner IChatClient, does its bit (before, after, or around the call), and delegates the rest inward. Stack several of those wrappers and you've built a middleware pipeline for model calls, exactly like the request pipeline in a web framework, but for the agent's connection to the model.

If every call goes through one interface, that interface is the place to add everything the call should also do.

Assembly

Building the Chain, Layer by Layer

A handful of fluent extension methods turn a bare client into a fully instrumented one.

The system collects these wrappers as fluent extension methods, each one a With* on IChatClient that returns a new IChatClient. Building a production-grade client is then just a readable chain. This is the code the agent pool runs when it constructs every agent.

// Read top to bottom as innermost to outermost. The first layer sits closest to
// the model; the last layer is the outermost shell that sees a request first.
IChatClient client = baseChatClient
    .WithRateLimitRetry()                  // retry throttled Azure OpenAI calls
    .WithAgent365Baggage(baggage)          // stamp Agent 365 identity telemetry
    .WithPurview(purviewOptions)           // apply Microsoft Purview governance
    .WithOpenTelemetry()                   // emit OpenTelemetry traces and metrics
    .WithCompaction(compactionProvider);   // trim history to fit the context window

Each method follows the same two-line shape: if the concern is configured, wrap; otherwise hand the inner client straight back. That second branch is important and we'll come back to it, but here's the form, using the governance layer as the example.

public static IChatClient WithPurview(this IChatClient inner, PurviewOptions? options)
{
    // Graceful no-op: if Purview isn't configured, return the inner client untouched.
    if (options is null || !options.Enabled)
        return inner;

    return new PurviewChatClient(inner, options);
}

The wrappers themselves derive from DelegatingChatClient, which forwards every call to the inner client by default, so each wrapper only overrides the one method it cares about and lets everything else pass through. The retry layer is the clearest example of how it catches one kind of failure and tries again.

internal sealed class RateLimitRetryingChatClient(IChatClient inner)
    : DelegatingChatClient(inner)
{
    public override async Task<ChatResponse> GetResponseAsync(
        IEnumerable<ChatMessage> messages, ChatOptions? options, CancellationToken ct)
    {
        for (var attempt = 0; ; attempt++)
        {
            try { return await base.GetResponseAsync(messages, options, ct); }
            catch (ClientResultException e) when (e.Status == 429 && attempt < MaxRetries)
            {
                // Honor the service's back-off hint, then try again.
                await Task.Delay(RetryAfter(e, attempt), ct);
            }
        }
    }
}
Order is behavior, not style

A request enters at the outermost layer and travels inward to the model, then the response unwinds back out. So compaction trims the history first (no point tracing or governing messages you're about to drop), the trace span wraps the governed-and-retried call so its timing is honest, and retry sits innermost, right against the model, so a throttled attempt is retried without re-running everything above it. Reorder the chain and you change what actually happens, not just how it reads.

The Layers

What Each Layer Actually Does

Five concerns, five wrappers, each one a real Microsoft capability snapped onto the same bridge.

This is the part where the concrete names matter, because each layer is a specific integration you'd want to know by name when you build your own. Here's the full stack, from the model outward.

WithRateLimitRetry

The RateLimitRetryingChatClient. Catches the throttling responses Azure OpenAI returns under load, waits for the service's suggested back-off, and retries. Sits innermost so a retry repeats only the model call, not the whole pipeline.

WithAgent365Baggage

The Agent365BaggageChatClient. Stamps Agent 365 identity telemetry onto every call: tenant, session, agent id, and agent name, built with a BaggageBuilder. The session is the pipeline's correlation id, so Agent 365 can group an entire event's activity.

WithPurview

The PurviewChatClient. Routes content through Microsoft Purview so the same data-governance and compliance policies that cover the rest of your estate cover what the agents send and receive. Unconfigured, it returns the inner client unchanged.

WithOpenTelemetry

Wraps the call in an OpenTelemetry span, emitting traces and metrics for each model interaction. Those flow through the standard exporters that AddServiceDefaults set up, so model calls show up in the same traces as everything else.

WithCompaction

Hooks in the Microsoft Agent Framework SDK's CompactionProvider to keep the conversation within the model's context window over a long, multi-turn run. It's the outermost layer.

One more job for baggage

The Agent 365 layer also tidies the message stream: it sanitizes orphaned function-call content that can otherwise make Azure OpenAI reject a request. That matters for the hosted-agent path below, where messages arrive shaped slightly differently.

The identity telemetry layer is worth one concrete look, because the baggage it builds is what ties the model-call view in Agent 365 back to the rest of the system. It's assembled from ids the pipeline already has on hand.

// Built once per agent and handed to .WithAgent365Baggage(...). The session id is
// the very same correlation id that threads intake, the bridge, and processing.
var baggage = new BaggageBuilder()
    .TenantId(tenantId)
    .SessionId(correlationId)
    .AgentId(agent.Id)
    .AgentName(agent.Name)
    .Build();
The Payoff

Optional by Design, the Same Code Everywhere

Because each layer disappears when unconfigured, one agent runs bare on your laptop and fully governed in production.

Remember that second branch in each extension method, the one that returns the inner client untouched when the concern isn't configured? That single decision is what makes the whole stack practical. Every layer is independently optional and degrades to a clean no-op. On your local developer environment with nothing configured, the chain collapses to the bare client and the agents run with no Purview, no Agent 365, no exporters. In production, the same chain lights up every layer. The agent code never changes; only the environment does.

It also explains how the optional hosted-agent path fits in without disturbing anything. For demonstration purposes, the research agent can be configured to run as a hosted Azure AI Foundry prompt agent instead of calling Azure OpenAI directly, constructed by turning a project client into an agent with AIProjectClient.AsAIAgent(new AgentReference(...)). Its instructions live server-side, but its model calls still flow through an IChatClient, so the exact same middleware chain wraps it. The Foundry agent gets the same retry, the same Agent 365 baggage, the same governance and tracing, with no special-casing. The Foundry agent isn't limited to the Researcher. Any of the agents could be from Foundry!

The reusable pattern

Find the one interface every important call passes through, then express each cross-cutting concern as an optional decorator over it. Compose them in a deliberate order, and make every layer degrade to a no-op when it isn't configured. You get production concerns without touching business logic, environment differences without code differences, and new concerns added by writing one more wrapper.

We've named the outermost layer but waved at what it does. Keeping a multi-turn, self-correcting conversation inside a fixed context window is a genuine problem that is solved with built-in compaction.
