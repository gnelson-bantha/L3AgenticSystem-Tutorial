Chapter 4

High Level Architecture

Every moving part on one map, and how a single event flows from left to right through the whole system.

Orientation

How to Read the Map

We've named the pieces in prose; now here they are as a picture you can actually click around in.

The last three chapters built up a vocabulary and a mental shape. This one turns that shape into something concrete. Below is the whole system as a left-to-right graph: work enters on the left, flows through the two stages and the data bridge between them, and finishes on the right. Arrows are labeled with the relationship, so you can read the diagram almost like a sentence.

A few of the boxes are system nodes, marked with a small "System" badge. Those are dense enough that drawing all their internals on the main map would turn it into spaghetti, so they're collapsed. Click one and it opens with its own little graph inside, showing just that subsystem's parts and how they connect. The intake service, the processing service, and the agent middleware stack are all collapsed this way. Everything else opens to a plain description. Go ahead and poke at the nodes as you read the walkthrough underneath.

The Big Picture

System Map

Click any node to expand it. Arrows show how each component relates to the next.

  • A Trigger submits an event to the Intake Service.
  • The Intake Service writes one work item per affected target to the Storage Bridge.
  • The Storage Bridge feeds parallel runs into the Processing Service.
  • Both the Intake Service and the Processing Service query data from the Tool Server over MCP.
  • Both services wrap each agent's model client with the Agent Middleware stack.
  • The Processing Service persists results to, and sends notifications through, State and Delivery.

Component

Triggers

The system doesn't care who starts it; it only cares that an event arrives. In the sample there are two front doors: an operator UI where a person submits a mock event and watches the agents work, a RESTful API endpoint, and a conversational front door (a Microsoft 365 Copilot declarative agent for demonstration purposes) that submits events. Either one posts the same event shape to the intake service.

Because the trigger is just "something posts an event," it's the easiest part to swap. An EventGrid, Service Bus, or continuous scanner that watches a real feed and posts events would slot in here without changing anything downstream.

System

Intake Service

The first stage is a short, fixed pipeline built as an explicit workflow graph. A request trigger receives the event and starts a run; a normalizer agent enriches it; a selector agent queries target data and writes one work item per match. Two agents, two nodes, one direction of flow.

Chapter 6 builds this graph node by node.

Component

Storage Bridge

A plain blob container is the entire contract between the two stages. Intake writes a work-item blob; processing polls the container and picks each one up. Neither service holds a reference to the other, which is what lets them scale, deploy, and recover independently. Like the Trigger, the Bridge can be replaced with another service that provides the same functionality (e.g. a database or a message queue).

This unglamorous box is doing the most important structural job in the system, so it gets a whole chapter of its own. See Chapter 8.

System

Processing Service

The second stage is where the bulk of the work happens. A poller watches the bridge and launches one run per work item. Each run is a collaboration runner that opens a Group Chat among three agents: a researcher gathers material, a generator writes the deliverable, and a reviewer either approves it or sends it back. The edge from reviewer back to generator is the self-correcting loop. Group Chat is one type of agentic workflow supported in the Microsoft Agent Framework SDK. Magentic workflows are another. The setup is a little different, but it is possible to change orchestration types to fit your use case.

Chapters 9 and 10 build the Group Chat and the loop that bounds it.

Component

Tool Server

Agents don't reach data directly. A standalone server exposes tools over the Model Context Protocol, and the agents call those tools without knowing where they run. The server offers data-query tools (look up targets by region, by size, by category) and a handful of workflow-control tools used by the conversational front door. A tool filter hides the control tools from the internal agents so they can't accidentally drive the workflow.

Because tools live behind a protocol boundary, you can add, change, or relocate them without touching agent code. This is largely present in the source repository and pattern to show how to leverage MCP servers with agentic workflows. Chapter 7 is all about this bridge.

System

Agent Middleware

None of the production concerns live inside the agents. Each one is a thin wrapper around the agent's model client, snapped on in a fluent chain. From the inside out: retry handling for rate limits, Agent 365 identity telemetry, Microsoft Purview content governance, OpenTelemetry tracing, and context compaction. Present or absent, each layer is optional, so the same agent runs bare locally and fully instrumented in production.

Chapters 11 and 12 take this stack apart layer by layer.

Component

State & Delivery

When a run finishes, its output has to land somewhere durable and the world has to hear about it. Finished deliverables go to a table store with a pointer blob; the conversation transcript for each run goes to a session store so a human can later re-open and re-run it; and a best-effort notification connector announces completion. All of it sits behind a small persistence layer so the agents never touch storage SDKs directly.

The session store is what makes the human-in-the-loop revision in Chapter 14 possible.

The Narrative

One Event, Left to Right

If you trace a single event through the boxes above, the whole design reads as one continuous motion.

Here's the life of one event in a paragraph. A trigger posts it to the intake service. Intake's normalizer agent cleans it up, then its selector agent asks the tool server which targets are affected and writes one work item per match into the storage bridge. That's the end of intake's job; it never waits for processing.

On the other side of the bridge, the processing service's poller notices the new work items and launches a separate run for each one, all at the same time. Inside each run, a researcher, a generator, and a reviewer take turns in a shared conversation. The reviewer holds the gate: it either approves the deliverable or sends it back to the generator, looping until it's satisfied or until a hard cap is reached. When the run finishes, the result is persisted to state and a notification goes out. Threaded through all of it, the same correlation id rides along so you can follow one event across both processes, and every agent's model calls pass through the middleware stack for retry, telemetry, governance, tracing, and compaction.

That's the entire system! This means you now have the background for the remainder of the tutorial. The pattern chapters each zoom into one box. We start where the program itself starts: the composition root, where every one of these pieces gets constructed and wired together. That's next.
