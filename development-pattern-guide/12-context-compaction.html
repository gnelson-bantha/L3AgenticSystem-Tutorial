Chapter 12

Context Management with a Compaction Pipeline

Keeping a long, multi-turn agent conversation from blowing past the model's context window.

The Squeeze

The Window Always Fills

A self-correcting loop is a conversation that grows, and the model can only hold so much.

Think about what a processing run actually accumulates. The research agent dumps in findings, sometimes large tool results. The generator writes a full draft, then another, then another as the reviewer sends it back. The reviewer adds critique each lap. By the time a hard case has gone three or four rounds, the shared transcript is long, and every model call has to fit inside the context window. A long autonomous loop is, by its nature, a conversation that grows without anyone pruning it.

You can't just let it overflow, and you can't just chop off the oldest messages either, because the early context (the brief, the key findings) is often what keeps the later turns coherent. What you want is to shrink the conversation thoughtfully: shed what's least valuable, preserve what carries meaning, and only get aggressive when you truly have to. The Agent Framework ships a mechanism for exactly this, and the system uses it as the outermost middleware layer from the last chapter.

Escalation

A Pipeline of Strategies, Cheapest First

Rather than one blunt rule, compaction is a ladder of trims that each fire only when needed.

The clever part is that compaction isn't a single tactic; it's a pipeline of strategies arranged from gentlest to bluntest, each guarded by its own threshold. They're composed with the framework's PipelineCompactionStrategy, which runs them in order and only applies the ones whose trigger has been crossed. On a short conversation, none of them fire. As it grows, they kick in one at a time, escalating only as far as the pressure demands.

// Ordered gentlest to bluntest. Each fires only when its own threshold is crossed,
// so a run escalates exactly as far as the conversation's size forces it to.
var strategy = new PipelineCompactionStrategy(
[
    // 1. Drop the bulky raw tool outputs once there are enough messages to spare them.
    new ToolResultCompactionStrategy    { MessagesExceed = 7 },

    // 2. Replace older turns with a running summary once tokens climb.
    new SummarizationCompactionStrategy { TokensExceed = 4096 },

    // 3. Keep only the most recent turns when the back-and-forth gets long.
    new SlidingWindowCompactionStrategy { TurnsExceed = 4 },

    // 4. Last resort: hard-truncate to fit before the window is exceeded.
    new TruncationCompactionStrategy    { TokensExceed = 32000 },
]);

Read the ladder and the philosophy is obvious. First it removes tool results, which are big and rarely needed verbatim once the agent has used them. If tokens still climb, it summarizes older turns, trading exact wording for compressed meaning. If the chat is still long, it slides a window over the recent turns and lets the older ones fall away. And only if all of that isn't enough does it hard-truncate, the bluntest move, reserved as a final guarantee that a call never exceeds the window. Each step is more lossy than the one before, which is exactly why each is tried only after the gentler ones came up short.

Tool-result trimming

The lightest touch. Verbose tool payloads are dropped once the conversation has enough messages, since the agent has already consumed what it needed from them.

Summarization

Older turns get folded into a compact summary as token counts rise, keeping the gist of the conversation while reclaiming a lot of room.

Sliding window

When the turn count itself grows large, only the most recent turns are kept in full, on the bet that recency carries most of the working context.

Truncation

The hard backstop. If everything else still leaves too much, the history is cut to fit. Lossy by design, last by intent, and a guarantee the call will fit.

Wiring

Plugging It In Per Agent

The strategy pipeline rides inside a provider that gives each agent its own compacted view.

The strategy pipeline is wrapped in a CompactionProvider, and that provider is what the .WithCompaction(...) layer from the previous chapter installs. Under the hood it registers as an AI context provider with UseAIContextProviders, keyed per agent. That per-agent keying matters: each participant in the Group Chat keeps and compacts its own view of the conversation, so the researcher's window and the generator's window are managed independently rather than sharing one pool.

// The provider carries the strategy pipeline. .WithCompaction(...) registers it as
// a per-agent AI context provider, so every agent manages its own context budget.
var compaction = new CompactionProvider(strategy);

IChatClient client = baseChatClient
    /* ...the other middleware layers... */
    .WithCompaction(compaction);   // outermost: trims before any inner layer runs
A heads-up about experimental APIs

The compaction types are marked experimental in the Agent Framework (they surface a diagnostic with the id MAAI001), which is the framework's way of saying the surface may still shift. You opt in deliberately by acknowledging that diagnostic where you use them. It's a small ceremony, but a useful signal: pin your versions and expect these particular APIs to evolve faster than the stable core.

The Lesson

Escalation Beats a Single Blunt Rule

The design generalizes well beyond context windows, which is why it's worth lingering on.

Step back from the specifics and there's a tidy principle here. When you have to give something up under pressure, don't reach for the most destructive option first. Arrange your responses from least costly to most costly, gate each behind a threshold, and let the situation decide how far up the ladder you climb. Most of the time the gentle moves are enough and the blunt ones never fire. When a genuinely hard case shows up, you still have a guaranteed backstop, but you reached it only after exhausting the kinder options.

That shape, ordered strategies with escalating cost and per-step triggers, shows up all over robust systems: cache eviction, load shedding, retry-and-degrade. Compaction is just a particularly clear instance of it, applied to the one resource an autonomous agent loop will always strain against. It's the last piece of the cross-cutting machinery.

With the agents fully equipped, we can zoom back out to the system level and answer a question we've been deferring: how do you follow one event across two separate processes, and how does the operator see progress while it all happens in the background? That's correlation and progress, and it's next.
