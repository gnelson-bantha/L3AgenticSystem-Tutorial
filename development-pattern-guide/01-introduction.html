Pattern Tutorial

Chapter 1

Introduction

What "Level 3" autonomy really means, and the shape of the pattern you're about to build.

The Idea

Autonomy at Level 3

Most AI features answer a question and stop. This one decides, acts, and finishes the job on its own.

Picture the difference between a chatbot and a colleague. A chatbot waits for you to ask something, answers, and waits again. You're the one holding the thread, deciding what happens next, stitching the steps together. A colleague is different. You hand them a goal, they go figure out the steps, they make judgment calls along the way, and they come back when the work is actually done. That second thing is what "Level 3" autonomy means, and it's the whole point of the system this guide teaches you to build.

It helps to put the levels side by side. At Level 1, the model is a smart autocomplete: it suggests, you decide. At Level 2, it can call tools, but a human still drives the loop and signs off on each move. At Level 3, the agent owns the loop. It reads the situation, picks the tools, judges the results, decides whether the work is good enough, and only hands back a finished product. No human sits in the middle of execution tapping "approve" between every step.

That sounds great until you try to build it, and then the hard questions show up all at once. If no human is in the loop, what stops an agent from confidently producing garbage? How do you run a hundred of these at the same time without them stepping on each other? How do you watch what they're doing in production when each one is making its own decisions? Those questions are exactly what the pattern in this guide answers, and none of the answers are "add a human checkpoint," because that would drop you right back to Level 2.

The working example

Throughout the guide we'll lean on one concrete scenario to keep things grounded: an event arrives describing some change in the world, and the system has to figure out which of many tracked entities are affected, then produce a reviewed written assessment for each one. It runs start to finish without anyone watching. The next chapter describes the real repository this is drawn from; everything after that generalizes the pattern so you can point it at your own problem.

The Pattern

The Shape of an Autonomous Pipeline

Here's the whole idea in one breath, before we spend sixteen chapters earning it.

Strip away the domain and the pattern is this: an autonomous, multi-stage agent pipeline whose stages are decoupled by durable storage, fanned out for parallelism, and kept honest by a bounded generate-and-review loop, with every cross-cutting concern bolted on as composable middleware rather than baked into the agents. That's a mouthful, so let's take it apart piece by piece, because each clause is a decision you'll be making in your own system.

Multi-stage. The work splits naturally into a cheap, fast intake phase (figure out what needs doing) and an expensive, slow processing phase (actually do it, well). Those two phases have completely different shapes and completely different scaling needs, so we run them as separate services instead of one monolith.

Decoupled by durable storage. Between the two phases sits a plain storage container. Intake writes work into it; processing reads work out of it. Neither one calls the other directly. That seam is what lets the two halves scale, deploy, and fail independently, and it's the difference between a demo and something that survives a crash.

Fanned out for parallelism. Intake doesn't produce one big job, it produces many small ones, one per affected entity. Each lands as its own item in storage, and the processing side picks them up and runs them at the same time. Ten affected entities means ten pipelines running in parallel, not one pipeline grinding through ten in a row.

Bounded generate-and-review. Inside the processing stage, the agents don't just produce output and trust it. One agent drafts, another reviews and can send it back for another pass. The loop self-corrects, which is how you get quality without a human gate. And it's bounded: there's a hard cap on how many times it can go around, so an agent that can never satisfy its reviewer still terminates instead of looping forever.

Composable middleware. Identity telemetry, content governance, distributed tracing, retry, context management: none of that lives in the agents. Each one is a thin wrapper around the agent's model client, snapped on like a Lego brick. You can run the exact same agent code bare on your laptop and fully instrumented in production, just by changing which wrappers are present.

Why this shape

Long-running autonomous work is unreliable as one in-process blob. Stages scale differently, agents drift without a quality gate, and you need governance and telemetry without rewriting agent logic. The pattern answers all three at once.

What it trades off

A storage seam buys independent scaling and crash recovery but adds eventual consistency and a little latency. A round-robin review loop is simpler than an explicit message graph but gives up fine-grained routing. A hard cap guarantees termination but can yield an unfinished draft you have to surface honestly.

What you get

True parallel processing, one codebase that runs bare locally and fully observable in production, and tool and model backends you can swap without touching orchestration. The cost is more moving parts and a storage contract you have to keep stable.

The Two Halves

Two Stages and One Storage Seam

Everything in the guide hangs off this one picture, so it's worth burning into memory now.

If you remember nothing else from this chapter, remember the seam. The system is two services that never speak to each other directly. They speak through a storage container, and that indirection is load-bearing.

The first service is the intake stage. It takes a raw incoming event, has an agent normalize and enrich it, then has a second agent query a data store to decide which entities the event actually affects. For each match, it writes one small work item into a blob container. That's its whole job: turn one vague event into a pile of concrete, independent units of work. It's a short, fixed pipeline, so we'll build it as a simple linear graph.

The second service is the processing stage. It watches that blob container, and every time a new work item shows up, it spins up an independent run to handle it. Each run is a small team of agents (a researcher, a writer, a reviewer) talking in a shared conversation until the reviewer signs off. When they're done, the result gets persisted and the world gets notified. Because each work item gets its own run, the processing stage is doing many of these at once.

The reason these are two separate services and not two functions in one process is the thing we keep coming back to: they're different animals. Intake is quick and runs once per event. Processing is slow, token-hungry, and runs once per affected entity. Both scale out, just on different triggers: a fresh intake instance is spawned for each incoming event, while the processing side runs as many instances as the backlog of work items demands. Splitting them at the storage seam is what makes that possible — you can scale the two halves independently, restart processing without touching intake, or replace the storage poll with a real event subscription later, all without renegotiating anything between the two. We'll spend Chapter 8 entirely on that seam because it earns the attention.

How the Rest of the Guide Is Laid Out

A quick tour of where we're headed, so each chapter lands in a slot you already have a place for.

The guide moves roughly in build order. First the framing: the next chapter walks the real repository this pattern came from (using its real names, the one place we do that), then prerequisites, then a clickable architecture map you can poke at.

After that come the pattern chapters, which are the heart of it. We start at the composition root, where the whole thing gets wired together, then build the intake stage, then the tool boundary that feeds the agents data over the Model Context Protocol. We cross the storage seam, build the group-chat processing stage, and add the self-correcting review loop that makes the output trustworthy. From there we get into the cross-cutting machinery: the middleware stack that adds Agent 365 identity telemetry, Microsoft Purview governance, and OpenTelemetry tracing around each agent; the compaction pipeline that keeps long conversations inside the context window; and the correlation id that lets you trace one event across two processes.

The last stretch is about living with the system: how a human can re-open a finished deliverable and re-run the same agents, how to extend the system with new agents and tools, and how to ship the whole thing to Azure with Aspire. Then an appendix of references for every piece of the stack.

One note before we start: from the next chapter on, the application's own names get generalized so the pattern stays portable. The Microsoft pieces it integrates with, Agent 365, Purview, Azure AI Foundry, the Agent Framework, MCP, Aspire, do not get generalized, because those integrations are a real part of the pattern and pretending otherwise would just make the guide vaguer and less useful. With that settled, let's look at the codebase this all comes from.
