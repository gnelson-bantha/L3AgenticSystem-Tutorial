Prerequisites

The tools, accounts, and background worth having in place before you build your own version of the pattern.

Starting Point

What You Should Already Be Comfortable With

You don't need to be an agent expert, but a few things will keep you from tripping in the first hour.

The last chapter named a lot of moving parts. This chapter makes sure you've got the ground under you before we start assembling them. The good news is the list is shorter than it looks, because Aspire and the Agent Framework do a lot of the heavy lifting.

On the language side, you want to be comfortable with modern C# and async/await. The whole system is asynchronous, agents stream tokens, storage calls are awaited, and background services loop on tasks. You should also be at ease with constructor dependency injection and the generic host, because that's how every service is wired. None of it is exotic, but if IServiceCollection and IHostedService are new to you, skim a primer first.

On the AI side, you only need a working mental model, not deep expertise. It helps to know that a large language model takes a list of messages and streams back a completion, that "tools" are functions the model can ask to call, and that an "agent" is just a model plus instructions plus (optionally) tools and memory. If you've ever wired up a chat completion call by hand, you're more than ready. You do not need prior experience with the Agent Framework, MCP, or Aspire; the point of the guide is to teach those.

Toolchain

What to Install

A .NET SDK, a container runtime for the local emulators, and an editor that understands Aspire.

The system targets .NET 10 and is orchestrated by Aspire, which ships as a project SDK rather than a separate workload now, so getting set up is mostly about the .NET SDK plus a container runtime for the local emulators. Here's the short list.

.NET 10 SDK

The whole solution targets net10.0. Install the SDK, then confirm with dotnet --version. Aspire comes in as the Aspire.AppHost.Sdk reference, so there's no separate workload to install.

A container runtime

Docker Desktop or Podman. Aspire uses it to run the Azure Storage emulator (Azurite) and the Cosmos DB emulator locally, so storage works on your laptop with no cloud account at all.

An Aspire-aware editor

Visual Studio, VS Code with the C# Dev Kit, or JetBrains Rider. Any of them can launch the Aspire AppHost and give you the dashboard that shows every service, log stream, and trace.

The Azure CLI

az is how you sign in for local development. The system uses your signed-in identity to reach Azure OpenAI, so a quick az login is part of the loop. No keys to copy around.

Once those are present, running the system is a single command from the AppHost project, which we'll get to in the deployment chapter. For now, just make sure the four boxes above are checked.

Credentials

The Model Backing and Why There Are No Keys

You'll need a model deployment to talk to, and the system reaches it through your identity rather than a secret.

The agents need a model to think with, and in the sample that's an Azure OpenAI chat deployment. You provide its endpoint and deployment name through configuration; everything else flows from your identity. There's a deliberate design choice hiding in that sentence, and it's worth pausing on because it shapes the whole codebase.

The system never uses an API key. Instead it authenticates with ChainedTokenCredential, which tries a sequence of credential sources in order and uses the first that works. Locally that's your az login session or your editor's signed-in account; in Azure it's the service's managed identity. The same code path works in both places, and there's no secret to leak, rotate, or accidentally commit.

Why this matters early

If you've built LLM apps with an API key in an environment variable, the keyless approach can feel like extra ceremony at first. It pays off immediately: the exact same agent code runs on your laptop under your own identity and in production under a managed identity, with role assignments (not secrets) deciding what each one can reach. Get az login working now and the rest of the guide just works.

Everything beyond the model is optional. Agent 365 telemetry, Microsoft Purview governance, Azure AI Foundry prompt agents, Microsoft Fabric data agents, and Logic Apps notifications all light up only when you configure them, and the system runs perfectly without any of them. That optionality is not an accident; it's the middleware pattern from Chapter 11 doing its job. You can build and run the whole thing against nothing but a model deployment and local emulators, then add the production concerns one at a time.

Vocabulary

A Five-Minute Agent Framework Primer

Five terms show up on nearly every page from here on, so let's define them once and move on.

The guide leans on a small vocabulary from the Microsoft Agent Framework. You'll pick most of it up in context, but seeding the definitions now means the architecture chapter reads smoothly instead of stopping to explain itself.

An IChatClient is the low-level handle to a model: give it messages, get back a streamed response. It's the seam everything else wraps around. A ChatClientAgent is an agent built on top of one: a model client plus instructions plus optional tools and memory, exposing a tidy "run this and give me the result" surface. The researcher, generator, and reviewer are all ChatClientAgent instances.

A workflow is how you connect more than one executor into a graph. The Agent Framework gives you WorkflowBuilder for explicit node-and-edge graphs (which intake uses) and AgentWorkflowBuilder for higher-level patterns like Group Chat, where several agents share one conversation and a manager decides who speaks next (which processing uses). Finally, MCP, the Model Context Protocol, is the standard way agents reach tools that live in a separate server; the agents call tools without knowing or caring where those tools run.

That's the whole glossary. With the toolchain installed, a model deployment in hand, and these five words defined, you're ready for the part everyone actually wants: the map of how it all fits together. The next chapter is a clickable architecture diagram you can explore node by node.
